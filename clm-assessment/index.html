Summary of my thoughts based on the conservation I had

  • I don't think we would be making use of all the capabilities offered by Pega CLM given that we are only modernizing our existing system and not building our Bank Onboarding from scratch.I am certain we would not be making use of the Product Designer and KYC frameworks.
  • Using the framework properly would definitely require a learning curve for our technical/Testing and Business teams. Business would definitely need to understand all the features of the framework thoroughly. The Requirements then would be more of filling in the Gaps or the functionality not provided by the framework. It is too early to know how big these Gaps could be. Also my past experience has taught me that when adequate efforts to learn the framework are not put by the Business and Technical teams then the framework just becomes a burden that we carry. The built application ends up not using many of the features provided by the framework. In some instances the framework could straightjacket us and due to that we end up not using features provided by the framework.
  • What if there are issues found in the framework. It would definitely take longer time to resolve such issues as that may require us to raise an SR and then follow it up with the Pega team.
  • It should also be considered that for building a generic framework such as CLM, Pega might have done some over-engineering which may affect the memory foot print of my application
  • To adhere to TOM Model we would need to make many customizations. The complexity of interfacing with OMDM and deciding the application path still needs to handled. Product Configuration and Negotiation flows where the application would spend maximum amount of its time would still need to be developed. Consuming and configuring service mappings would still need to be done. Etc Etc
  • Developers would need to thoroughly understand the framework Data Model and comply with that. Sometimes creating a new data model might be easier.
  • No doubt the framework is good, the framework would guide me on Case Design and class structure to some extent. Requirements feature is appealing but the number of cases created and existence of already built Content Management Systems needs to be factored in. The User Experience is indeed excellent; especially the Dashboard and the Review/Confirm Harnesses. So there are definitely some very appealing and exciting features. But does that warrant the use of this FrameWork ???
  • As an alternative, may be we can work on the eIAP framework which is already aligned with the TOM Model to some extent. Too bad that we did not work on lending extensively else it would have completely aligned with the TOM Model. Further we can work on the following items in eIAFW
    1. Introduce a Customer Interview to better assess his needs
    2. Make improvements to the UI design and test it on Tabs and Mobiles
    3. Make more skeleton rules, stages and steps to serve as a guide for future development
    4. Have Brainstorming sessions and incorporate more ideas and feedback
